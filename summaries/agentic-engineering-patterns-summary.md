# Agentic Engineering Patterns

> Treat a coding agent as a fast, tireless pair-programmer that must *prove* its work — feed it reusable reference code and clear tests, let it run/verify/iterate via tools and subagents, and bank your learnings back into its instructions.

## Context

A practical summary of Simon Willison's [Agentic Engineering Patterns](https://simonwillison.net/guides/agentic-engineering-patterns/) guide, organized as a workflow for using an LLM / coding agent effectively in a real coding project.

The mental model: a coding agent = an **LLM wrapped in a harness** that lets it *run tools in a loop* (read files, execute code, run tests, call the browser). The key advantage over plain chat is **execution** — the agent can prove its code works instead of guessing.

> **Hard rule:** "Never assume code generated by an LLM works until it has been executed."

This is **agentic engineering**, not vibe coding:
- **Vibe coding** = accept unreviewed prototype output and move on.
- **Agentic engineering** = deliberately direct the agent, verify, and iterate to production quality.

You (the human) still own the hard parts: defining *what* to build, choosing among tradeoffs, and verifying the result.

## Key points

- **The economic shift:** writing code is now cheap, but *good* code (correctness, tests, maintainability, security, accessibility, solving the right problem) still costs. Revisit ideas you dismissed as "not worth the time," and run experimental prompts in parallel low-stakes sessions. *"Shipping worse code with agents is a choice."*
- **Build a personal "hoard"** of working solutions and point the agent at reference repos instead of explaining logic in prose.
- **Prompt well:** name established tools/libraries directly, and give the agent a way to test itself mid-task so it self-corrects.
- **Lead with tests** — start sessions with "first run the tests"; use red/green TDD and agentic manual testing.
- **Use subagents** to manage context (explore, parallel edits, specialists) — but don't over-fragment.
- **Use Git as a safety net** and let the agent drive it with plain-English prompts.
- **Understand the code you generate** to fight "cognitive debt."
- **Don't file PRs with unreviewed code.**
- **Play the long game:** fold each project's lessons back into your agent instructions/tools.

## Details

### 1. Build (and reuse) a personal "hoard"
Keep a collection of working solutions — blog posts, TIL notes, GitHub repos, small tools. Agents can search these and *recombine* them (e.g. merging PDF.js + Tesseract.js into an OCR tool in minutes). Point the agent at reference repos rather than explaining logic in prose:

> "Clone `simonw/blog` to /tmp for reference, mirror how X already works."

### 2. Prompt well — assume broad knowledge, add testing hooks
- Name established tools/libraries directly (Gifsicle, Emscripten, Playwright) — the agent knows them.
- Give it a **way to test itself mid-task** (a test URL, a dev server, a browser tool). This lets it self-correct and catch its own bugs instead of handing you broken code.
- Keep prompts focused; let the agent explore the codebase for details.
- Iterate with follow-up prompts (core feature first, then infra / licensing / polish).
- Lean on agents for tedious trial-and-error humans give up on (like WASM compilation).

### 3. Lead with tests
- **"First run the tests"** at the start of a session — forces the agent to find and read your tests, learn the codebase, and get into a testing mindset.
- Use **red/green TDD**: write tests first, confirm they *fail*, then implement until green. Modern models understand "use red/green TDD" as shorthand.
- **Agentic manual testing** for what unit tests miss:
  - `python -c "..."` for quick function checks
  - temp scripts in `/tmp` (avoid accidental commits)
  - `curl` against a dev server for web APIs
  - **browser automation** (Playwright / agent-browser / Rodney) for web UIs
- Tools like **Showboat** record commands + real output + screenshots, so testing becomes trustworthy documentation.

### 4. Use subagents to manage context
- **Explore subagent** maps the codebase and returns findings without burning the main context window.
- **Parallel subagents** for editing independent files simultaneously (can use cheaper/faster models like Haiku).
- **Specialist subagents**: code reviewer, test runner, debugger.
- Don't over-fragment — the main agent can do these too.

### 5. Use Git as a safety net — and let the agent drive it
Plain-English prompts work:
- "commit these changes"
- "integrate latest from main"
- "sort out this git mess"
- "use git bisect to find when this bug was introduced"

Agents are great at nasty merge conflicts, `reflog` recovery, and rewriting history (squash, split, extract-to-new-repo preserving authorship).

### 6. Understand the code you generate — fight "cognitive debt"
Don't let agent code become a black box:
- **Linear walkthroughs**: have the agent read the repo and produce a walkthrough using real code snippets (via `sed`/`grep`/`cat`, not memory, to avoid hallucination).
- **Interactive explanations**: have it build an animated HTML visualization/simulation of an algorithm (e.g. word-cloud spiral placement) instead of relying on jargon.

### Anti-patterns to avoid
- **Don't file PRs with code you haven't reviewed yourself.** Dumping thousands of unreviewed lines just offloads your job onto reviewers.
- **Good agent-assisted PRs are:** verified working; small / scoped; accompanied by clear context and linked issues; given a description you actually vetted; backed by evidence (test notes, screenshots, video).

### The long game: the compound engineering loop
After each project, run a retrospective, capture what worked, and **fold those lessons back into your agent instructions/tools** — because LLMs don't learn from their own past mistakes; *you* have to encode the learning. Over time this compounds into steadily better results.
